# Zachary Muon Clusters

An 8-detector plastic scintillator array built at Muhlenberg College to detect
cosmic ray air shower coincidences. Eight Arduino Nano microcontrollers each
record the arrival time of every muon passing through their detector. The goal
is to test whether the detectors fire independently (the null hypothesis) or
whether multi-detector coincidences occur significantly more often than chance
— which would indicate that a single air shower deposited secondary particles
in multiple detectors nearly simultaneously. The February 27, 2026 run
produced a statistically significant excess at the ~4–5σ level.

## Analysis Workflow

### Running the analysis on existing data

From the repository root:

```bash
python muon_analysis.py               # Feb 27 baseline
python muon_analysis_jump_veto.py     # Feb 27 with jump-interval veto
```

Plots are written automatically to `runs/feb27_GOOD/plots/`.

### Setting up a new run

1. Create a new run directory and a `plots/` subdirectory inside it:
   ```
   runs/RUNNAME_DATE/
   runs/RUNNAME_DATE/plots/
   ```

2. Copy the raw CSV file from the Arduino serial log into that directory.

3. Choose the appropriate script:
   - Phase 01 hardware (10s sync intervals): use `muon_analysis.py` or
     `muon_analysis_jump_veto.py`
   - Phase 02 hardware (1s sync intervals, 2-hour run): use
     `muon_analysis_2hr.py`

4. Update `DATA_FILE` near the top of the chosen script to point at the new
   CSV. `PLOT_DIR` is derived automatically — plots will be written next to
   the data.

5. Run the script. All 12 output plots (6 main + 6 diagnostic) will appear
   in `runs/RUNNAME_DATE/plots/`.

### What each script does

| Script | Input | Key addition |
|--------|-------|--------------|
| `muon_analysis.py` | `runs/feb27_GOOD/data_acquisition_2-27.csv` | Baseline pipeline |
| `muon_analysis_jump_veto.py` | Same | Adds veto for I2C interrupt-deferral artifacts in sync timestamps |
| `muon_analysis_2hr.py` | `runs/mar14_FAILED/data_acquisition_3-14.csv` | Configured for 1s sync intervals and longer runs |

All three scripts share the same 8-step pipeline:
1. Parse CSV → PT (pulse) and ST (sync) events
2. Build sync table → `sync_array[k, i]` = Nano i's local time at sync k
3. Build offset table → `offset[k, i] = sync_array[k, i] − sync_array[k, 0]`
4. Calibrate PT timestamps via piecewise-linear interpolation
5. Apply sync veto (±20 ms around each ST event)
6. Find coincidences: greedy sliding window, earliest-first
7. Compute expected accidentals: `E_k = T × Σ Π r_i × (2τ)^(k−1)`
8. Compute significance: `σ = Φ⁻¹(1 − P(X ≥ observed | Poisson(expected)))`

See `docs/analysis_guide.md` for a detailed plain-English description of every step.

## Current Status

### Feb 27, 2026 run — COMPLETE AND VALIDATED

| Quantity | Value |
|---|---|
| Run duration | ~59.9 min |
| Total PT events | 28,935 |
| Events after sync veto | 28,798 |
| Per-detector rates | 0.81–1.25 Hz |
| 2-fold @ 1 ms | 274 obs / 202 exp / **+4.8σ** |
| 3-fold @ 1 ms | 7 obs / 0.8 exp / **+4.1σ** |
| 5-fold @ 1 ms | 1 obs / ~0 exp / **+4.5σ** |
| Best candidate | 5-fold at 49.97 min, span = 3.6 µs, Nanos {1,3,5,6,7} |

Results are robust: confirmed by null test (200 time-shift trials) and stable
under the jump-interval veto (≤ 0.2σ change). Below the conventional 5σ
discovery threshold, but consistent with genuine air shower detections.

### Mar 14, 2026 run — FAILED (data unusable)

The Phase 02 firmware uses `INPUT` instead of `INPUT_PULLUP` on the sync
pin. At 1-second sync intervals, this causes bounce/double-triggering within
the 5 ms debounce window, producing alternating ~480 ms / ~580 ms intervals
instead of ~1000 ms. The jump-interval veto correctly identifies ~68% of
intervals as corrupted; significance drops to ~2σ. See
`runs/mar14_FAILED/FAILED_RUN_NOTES.md` for the root-cause analysis and the
one-line firmware fix required.

### Next step

Fix the `INPUT → INPUT_PULLUP` bug in
`firmware/phase_02/FOLLOWER-NANO_phase02.ino`, reflash all 8 Nanos, and
take a new 2-hour run.

## Hardware

- 8× Arduino Nano (followers) — one per detector
- 1× Arduino Uno (master) — broadcasts sync pulses, collects data via I2C
- Plastic scintillator tiles with SiPM readout
- Shared sync wire connecting all Nanos and Uno

Firmware in `firmware/phase_01/` was used for the Feb 27 run.
Firmware in `firmware/phase_02/` is the improved version intended for
2-hour runs with 1-second sync intervals and hardware-timer-based timestamp
capture (zero software latency for particle pulses).
